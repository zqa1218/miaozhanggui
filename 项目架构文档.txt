喵掌柜自助下单系统 — 项目架构文档 v1.0
============================================================

一、系统概览
-----------

喵掌柜是一套面向小型摄影/妆娘/棚主的多角色预约接单平台。
商家通过 B端管理后台创建项目、配置时间轴与定价，
客户通过 C端浏览项目、选择服务时段并支付定金完成预约。

核心业务流程：
  项目上架 → 时间轴配置 → 客户预约 → 定金锁定 → 商家确认 → 服务完成 → 尾款结清

五、请求处理分层
--------------

  HTTP Request
    → Middleware Chain (helmet → cors → morgan → rateLimiter → deviceId → auth → validator)
    → Controller (req 解析 → 调用 service → res.rh 响应)
    → Service (业务逻辑 + 事务编排 + DTO 映射)
    → Repository (knex SQL 查询)
    → MySQL / Redis

每个模块遵循 routes → controller → service → repository 分层：
  - Routes：定义 HTTP 方法、路径、中间件链
  - Controller：解析请求、调用 service、通过 res.rh 统一响应
  - Service：核心业务逻辑、事务编排、权限校验
  - Repository：纯 SQL 查询封装，不包含业务逻辑
  - Validator：Joi schema 定义，在路由层通过中间件执行


六、共享基础设施
--------------

1. 统一响应格式 (res.rh)
   res.rh.success(data, msg) → { success: true, code: 0, data, message }
   res.rh.fail(msg, status)   → { success: false, code: 1, message }
   res.rh.error(msg)          → { success: false, code: 1, message } + 500

2. 错误体系 (AppError + errorCodes)
   所有业务异常抛出 new AppError(ERROR_CODES.STUDIO_NOT_FOUND, 404)
   由 errorHandler 中间件统一捕获
   分为 isOperational=true（预期内）和 unexpected 两种

3. WebSocket 实时通知
   订单创建后通过 WebSocket 向对应 m_id 的商家推送
   B端实时感知新订单

4. 双锁时段机制
   pre_lock（第一重）：客户支付定金后自动创建，标记该时段不可被其他客户预约
   hard_lock（第二重）：商家确认收款后升级，正式排入工作日历

十、订单状态机
------------

  客户下单 ──→ 待支付(定金待确认)
                  │
      商家确认 ←──┘ ──→ 已付定金 ──→ 已确认锁定(hard_lock)
                  │        │
            服务完成 ↓        │
                  │        ↓
            尾款待确认 ──→ 已结清 ──→ 已完成拍摄

    申请取消/改期 ←── 任意未完成状态
                  │
      商家审批 ←──┘ ──→ 已取消 / 已退款取消


十一、部署运维
------------

  【开发环境】 bash运行环境/start.sh 一键启动
    1. docker compose up -d（MySQL + Redis）
    2. npm install（后端 + 前端依赖）
    3. npx knex migrate:latest（数据库迁移）
    4. node src/server.js（后端 :3000）
    5. npx vite（前端 :5173）

  【生产部署】 npm run prod
    pm2 start ecosystem.config.js（Cluster 模式，零停机 reload）

  【数据库迁移】
    当前最新迁移：038_add_address_fields.js
    迁移脚本均为幂等设计（先检查列/表是否存在再操作）
