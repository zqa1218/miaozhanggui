## 一、当前数据库表结构 (10 张表)

### 1.1 merchants — 商家表

| shop_name | VARCHAR(128) | NOT NULL | 店铺名称 |

## 二、当前架构的核心问题

### 2.1 客户端无登录体系
- 当前 C 端通过 `user_device_id`（localStorage 随机ID）识别用户
- 无账号系统，换设备/清缓存后订单无法找回
- 无法做用户画像、积分、会员等运营功能

### 2.2 商家端仅支持用户名密码登录
- 无微信小程序登录能力
- `shop_mode` 仅区分 `studio` / `photographer` 两种
- 无法支撑"摄影 / 棚主 / 妆造 / 假发"四角色独立运营

### 2.3 数据无角色隔离
- 所有商家数据通过 `m_id` 关联，但无 `merchant_type` 区分
- 不同角色商户的数据逻辑上混在一起，仅靠前端过滤不可靠

### 3.3 四角色数据隔离方案

```
┌─────────────────────────────────────────────────────────────────┐
│                    数据隔离架构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  核心原则：所有业务表通过 m_id + merchant_type 双重过滤           │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  merchants 表 (角色定义层)                                │    │
│  │  ┌──────────┬──────────────┬───────────────────────┐    │    │
│  │  │ m_id     │ merchant_type│ shop_name             │    │    │
│  │  ├──────────┼──────────────┼───────────────────────┤    │    │
│  │  │ shop_001 │ photographer │ 张三摄影工作室         │    │    │
│  │  │ shop_002 │ studio_owner │ 阳光摄影棚             │    │    │
│  │  │ shop_003 │ makeup_artist│ 小花妆造               │    │    │
│  │  │ shop_004 │ wig_stylist  │ 假发专家-李四          │    │    │
│  │  └──────────┴──────────────┴───────────────────────┘    │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
│  查询隔离方式（推荐方式 B —— 性能更优）：                         │
│                                                                  │
│  方式 A：JOIN 过滤                                               │
│  SELECT o.* FROM orders o                                       │
│  JOIN merchants m ON o.m_id = m.m_id                            │
│  WHERE m.merchant_type = 'photographer'                         │
│                                                                  │
│  方式 B：冗余 merchant_type 到业务表 ★推荐                       │
│  — 在 orders/studios 等表增加 merchant_type 字段                  │
│  — 查询时直接 WHERE merchant_type = 'xxx'                        │
│  — 避免 JOIN，性能更好                                           │
│  — 缺点：需要保证数据一致性                                       │
│                                                                  │
│  方式 C：JWT Token 携带角色信息                                   │
│  — 登录时 JWT payload 包含 { mId, merchantType }                 │
│  — 各模块查询时自动附加 merchant_type 过滤条件                    │
│  — 配合 auth 中间件自动注入上下文                                 │
│                                                                  │
│  ★ 最终推荐：A + C 组合                                          │
│  — JWT 携带 merchant_type                                        │
│  — 中间件将 { mId, merchantType } 注入 req.user                  │
│  — 各模块按需 JOIN 或直接 WHERE 过滤                              │
│  — 无需冗余字段，数据一致性由 merchants 表保证                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 登录流程设计

#### C端（客户端）微信小程序登录流程

```
  微信小程序                        后端服务器                     数据库
  ─────────                        ─────────                     ──────
     │                                 │                            │
     │  ① wx.login() 获取 code         │                            │
     │────────────────────────────────→│                            │
     │                                 │                            │
     │                                 │  ② code2Session             │
     │                                 │  (换取 openid + session_key)│
     │                                 │                            │
```

## 六、关键注意事项

1. **微信 openid 安全**：openid 和 session_key 属于敏感数据，不可暴露给前端，必须后端中转
2. **session_key 管理**：微信 session_key 有过期时间，需定期刷新；不可存入 JWT
3. **unionid 跨应用**：如果需要在多个微信应用（公众号+小程序）中识别同一用户，需要使用 unionid（需绑定微信开放平台）
4. **多角色绑定**：一个微信 openid 可绑定多个商家角色（如一个人既是摄影师也是妆造师），登录时需展示选择界面
5. **数据隔离**：四角色虽然共用同一套表结构，但通过 `merchant_type` + `m_id` 实现逻辑隔离，确保摄影师的订单不会出现在棚主后台
6. **灰度迁移**：建议先在 `shop_mode` 为 `studio` 的商家中测试新 `merchant_type` 功能，平稳后再全量切换
