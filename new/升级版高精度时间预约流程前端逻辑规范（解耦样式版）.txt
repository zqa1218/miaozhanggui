================================================================================
  第一部分：商家端「样式独立创立流程」 (B端 — 优先于项目创立)
================================================================================

【独立窗口 0：商家独立样式创立与管理窗口 (Window_B_Style_Create)】
  - 业务流向：商家在进入项目创立前，在独立的样式库中优先创立接单样式资产。
  - 前端输入信息：
    * styleName          : 样式名称 (String)
    * styleCoverUrl      : 风格示意图 (String, 单图上传组件获得)
    * singlePrice        : 该样式单张收费金额 (Decimal)
    * hasPackage         : 该样式是否提供套餐收费 (Boolean 开关)
    * packagePrice       : 该样式套餐收费金额 (Decimal, 仅在 hasPackage 为 true 时必填)
  - 动作按钮：[保存样式] -> 触发 POST /api/styles 接口，将独立样式资产存入通用样式库。

================================================================================
  第二部分：商家端「创立项目」完整流程与输入信息 (B端 — 3个顺序跳转窗口)
================================================================================

【窗口 1：项目基础初始化窗口 (Window_B_Step1)】
  - 业务流向：完成项目物理层面的基本盘收集[cite: 1]。
  - 前端输入信息：
    * title              : 项目名称 (String)[cite: 1]
    * city               : 地点城市 (String)
    * availableDates[]   : 可选日期数组 (Array of DateStrings, YYYY-MM-DD)
  - 跳转触发：输入项非空校验通过，点击 [下一步]，当前窗口销毁，激活并跳转至 Window_B_Step2。

【窗口 2：高精度时间轴与休息扣除配置窗口 (Window_B_Step2)】
  - 业务流向：承接窗口 1 的日期数组，配置每日不间断工作总区间及非接单扣除时段[cite: 1]。
  - 前端输入信息：
    * baseStartTime      : 基础可接单工作起始时间 (TimeStr, 如 "09:00")
    * baseEndTime        : 基础可接单工作结束时间 (TimeStr, 如 "21:00")
    * restSlotCount      : 休息分段数 (Integer)
    * restSlots[]        : 休息分段起止时间数组 (结构为 Array of {start_time, end_time})
    * intervalRestTime   : 每单结束后间隔休息时间 (Integer, 单位：分钟)
    * isReplicated       : 复刻标志位 (Boolean, 勾选代表将该时间规则应用到窗口 1 的所有可选日期)
  - 跳转触发：点击 [下一步]，当前窗口销毁，激活并跳转至 Window_B_Step3。

【窗口 3：计价模式选择与动态时长配置窗口 (Window_B_Step3)】
  - 业务流向：决定项目是否使用样式功能，并根据选择进行定价权分流及服务耗时定义。
  - 前端输入信息与控制逻辑：
    * isStyleEnabled     : 是否使用样式功能 (Boolean 开关组件)
    
    * [分流场景 A：isStyleEnabled == true (使用样式功能)]
      ├─ styleSelector   : 前端调用 GET /api/styles 接口拉取先前创立好的样式库。
      └─ selectedStyleIds: 样式多选勾选框，商家自选哪些样式可以“出现在本项目中供人选择”。
      *(注：此时隐藏并禁用项目原生单张/套餐计价输入框，直接沿用所选样式的收费标准)*

    * [分流场景 B：isStyleEnabled == false (不使用样式功能)]
      ├─ singlePrice     : 项目原生单张收费价格 (Decimal)
      ├─ hasPackage      : 项目原生是否加入套餐 (Boolean 开关)
      └─ packagePrice    : 项目原生套餐收费价格 (Decimal, 仅在 hasPackage 为 true 时可见)

    * [通用服务时间与加时配置 (所有场景必填)]：
      ├─ singleShotTime     : 单张平均拍摄时间 (Integer, 单位：分钟)
      ├─ packageTime        : 套餐标准拍摄时间 (Integer, 单位：分钟)
      ├─ isExperienceOptionEnabled : 是否开启模特经验选择 (Boolean)
      ├─ noviceSingleAddTime: 新人单张加时时间 (Integer, 单位：分钟)
      ├─ novicePackageAddTime: 新人套餐加时时间 (Integer, 单位：分钟)
      └─ depositRatio       : 定金比例 (Integer, 5/10/30/50/100)[cite: 1]
  - 终止触发：点击 [完成并上架项目]，触发 POST /api/add-studio 打包提交，B端项目创立流程闭环。

================================================================================
  第四部分：客户端「下订单」完整逻辑与输入信息 (C端 — 2个独立顺序窗口)
================================================================================

【窗口 4：客户服务选择与时间段预估精算窗口 (Window_C_Step1)】
  - 业务流向：客户进入页面，系统判断项目是否启用样式。客户选择SKU及起始时间，前端精算时长[cite: 1]。
  - 前端渲染与条件分流逻辑：
    1. 页面加载时请求项目详情。若项目 `is_style_enabled == true`，则界面强制渲染 [选择拍摄样式] 组件（数据源来自 `studio_style_relations` 过滤出的样式），客户必须选择一个样式[cite: 1]。
    2. 价格源获取分流：
       - 若使用样式：系统的计价标准转而读取所选样式下的 `single_price` 或 `package_price`。
       - 若不使用样式：界面隐藏样式选择组件，直接读取项目主表（`studios`）自带的 `single_price` 或 `package_price`。
  - 前端输入信息：
    * selectedStyleId    : 选中的样式 ID (若项目启用样式功能则必填，否则为 Null)
    * optType            : 预约服务类型 (Enum: 'single' 按单张 / 'package' 选套餐)
    * photoCount         : 拍摄张数 (仅在 optType == 'single' 时可见)
    * modelExperience    : 模特经验属性 (Enum: 'newcomer' 新人 / 'experienced' 老手，由商家端决定是否渲染)
    * bookingStartDate   : 拍摄预约日期 (DateStr)
    * bookingStartTime   : 拍摄起始时间点 (TimeStr, 如 "14:00")
  - 前端动态时间精算与阻断逻辑：
    - 根据输入实时演算：TotalDuration = (单张/套餐基础时间) + (经验加时) + interval_rest_time[cite: 1]。
    - bookingEndTime = bookingStartTime + TotalDuration[cite: 1]。
    - 展示看板：【您的预计拍摄时间段为：bookingStartTime 至 bookingEndTime】[cite: 1]。
    - 自动碰撞校验：比对 `slot_bookings`（已锁定订单）与 `studio_rest_slots`（商家休息段），冲突则红字报错并强行置灰 [下一步] 按钮[cite: 1]。
  - 跳转触发：时间段完全可用，点击 [下一步]，当前窗口销毁，激活并跳转至 Window_C_Step2。

【窗口 5：角色信息完善与双重时间锁死提交窗口 (Window_C_Step2)】
  - 业务流向：录入业务文本信息，通过“定金锁 + 确认锁”的双锁机制完成时间块的排他性占领[cite: 1]。
  - 前端输入信息：
    * roleName           : 想要拍摄的角色名称 (String, 强制必填)
    * contactNote        : 联系方式及特殊拍摄备注 (String, 选填)[cite: 1]
  - 双重时间锁定控制流转逻辑：
    - 【第一重锁：用户端定金锁】
      顾客点击提交订单并支付定金[cite: 1]。触发 POST /api/pay-deposit 接口后，后端将排期表锁状态置为 `lock_type = 'pre_lock'`[cite: 1]。此时【顾客端的预定时间段被锁死】，C端视图对其他所有人置灰该时间段，防并发抢单[cite: 1]。
    - 【第二重锁：摄影师确认锁】
      摄影师在B端后台待办列表审查订单的精确时间段、样式、角色名称和备注，手动点击 [确认接单][cite: 1]。接口 POST /api/order/confirm-lock 触发，后端将状态更新为 `lock_type = 'hard_lock'`【彻底锁死订单时间】[cite: 1]，正式加入摄影师的订单工作列表[cite: 1]。
