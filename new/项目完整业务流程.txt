================================================================================
  喵喵自助下单平台 — 完整业务流程
  从建立项目 → 用户预约 → 支付 → 拍摄完成 → 结清尾款
================================================================================

  平台角色说明：
    B端 = 商家管理后台 (admin.html)，需 JWT 登录
    C端 = 用户预约端 (index.html)，通过 URL 参数 ?mId=xxx 访问，无需登录

================================================================================
  第一阶段：商家入驻与项目上架 (B端)
================================================================================

步骤 1.1  商家注册
  ─────────────────
  接口：POST /api/register
  需填写信息：
    · username      — 登录账号 (3~32字符)
    · password      — 登录密码 (6~64字符)
    · shopName      — 店铺名称 (1~128字符)
  后端处理：
    · 检查 username 是否已存在
    · 生成 m_id (格式: shop_${时间戳})
    · bcrypt 哈希密码 → password_hash
    · 写入 merchants 表
    · 返回 JWT Token + mId + shopName
  ┌──────────────┐
  │ merchants 表  │  id / m_id / username / password_hash / shop_name /
  │              │  shop_mode('studio'|'photographer') / qr_code_url /
  │              │  alipay_qr_url / wechat_qr_url / announcement /
  │              │  created_at / updated_at
  └──────────────┘

步骤 1.2  商家登录
  ─────────────────
  接口：POST /api/login
  需填写信息：
    · username — 登录账号
    · password — 登录密码
  后端处理：
    · 查询 merchants 表按 username
    · bcrypt.compare 验证密码
    · 签发 JWT Token (payload: { mId, username }, 有效期 2h)
  前端存储：
    · localStorage: mzg_admin_token (JWT)
    · localStorage: mzg_admin_mid  (商家 m_id)

步骤 1.3  上架新项目 (核心配置步骤)
  ─────────────────
  接口：POST /api/add-studio  (需 JWT 认证)
  ┌─────────────────────────────────────────────────────────────────┐
  │                        需要配置的信息                             │
  ├─────────────────────────────────────────────────────────────────┤
  │ 基础信息：                                                       │
  │   · title              — 项目名称 (如 "Coser摄影体验")            │
  │   · description        — 项目描述文字                            │
  │   · coverUrl           — 封面图片URL (上传获得)                   │
  │   · detailImgUrls[]    — 详情图URL数组 (上传获得)                 │
  │   · serviceMode        — 服务模式: 'studio'(工作室) |            │
  │                                    'photographer'(摄影师)       │
  ├─────────────────────────────────────────────────────────────────┤
  │ 计费设置：                                                       │
  │   · calcMode           — 计费方式:                              │
  │       'time'             = 按时段计费 (时段数 × 单价)             │
  │       'photo'            = 按张数计费 (张数 × 单价)               │
  │       'photo_time'       = 按张+时段 (每时段限制最大张数)         │
  │   · isHolidayPrice     — 是否节假日不同价 (true/false)           │
  │   · price              — 统一价格 (元)                           │
  │   · weekdayPrice       — 工作日价格 (启用节假日不同价时有效)      │
  │   · holidayPrice       — 节假日价格 (启用节假日不同价时有效)      │
  ├─────────────────────────────────────────────────────────────────┤
  │ 包天设置：                                                       │
  │   · isPackageDay       — 是否支持包天 (true/false)               │
  │   · packagePrice       — 包天总价 (元)                           │
  ├─────────────────────────────────────────────────────────────────┤
  │ 折扣设置：                                                       │
  │   · hasDiscount        — 是否开启多时段折扣 (true/false)         │
  │   · discountRules[]    — 折扣规则: [{hours: 满N小时, price: 单价}]│
  │       例: [{hours:3, price:80}, {hours:5, price:70}]            │
  │       含义: 满3小时→80元/时, 满5小时→70元/时                      │
  ├─────────────────────────────────────────────────────────────────┤
  │ 时间与排期：                                                     │
  │   · openTime           — 营业开始时间 (默认 09:00)               │
  │   · closeTime          — 营业结束时间 (默认 21:00)               │
  │   · timeInterval       — 排期间隔(分钟) (默认 60, 即每小时一个时段)│
  ├─────────────────────────────────────────────────────────────────┤
  │ 张数与人数：                                                     │
  │   · minPhotos          — 起订张数 (默认 1)                       │
  │   · slotMaxPhotos      — 每时段最大接单张数 (photo_time 模式)    │
  │   · extraPersonFee     — 额外每人费用 (元, 第2人起)              │
  ├─────────────────────────────────────────────────────────────────┤
  │ 定金设置：                                                       │
  │   · depositRatio       — 定金比例: 5 / 10 / 30 / 50 / 100 (%)   │
  ├─────────────────────────────────────────────────────────────────┤
  │ 其他：                                                           │
  │   · isFieldPhoto       — 是否场照接单                            │
  │   · fieldPhotoDates[]  — 场照日期数组                            │
  │   · customRemarkLabel  — 自定义备注标签 (如 "CN" / "角色名")     │
  └─────────────────────────────────────────────────────────────────┘
  写入表：studios
  ┌──────────────┐
  │ studios 表    │  id / m_id / title / description / cover_url /
  │              │  detail_img_urls(JSON) / service_mode / calc_mode /
  │              │  is_holiday_price / price / weekday_price /
  │              │  holiday_price / is_package_day / package_price /
  │              │  has_discount / discount_rules(JSON) / deposit_ratio /
  │              │  open_time / close_time / time_interval /
  │              │  min_photos / slot_max_photos / extra_person_fee /
  │              │  is_field_photo / field_photo_dates(JSON) /
  │              │  custom_remark_label / is_deleted /
  │              │  created_at / updated_at
  └──────────────┘

步骤 1.4  设置不可接单时段 (可选)
  ─────────────────
  接口：GET  /api/unavailable-slots/grid    — 查询某日不可接单时段网格
        POST /api/unavailable-slots/save    — 保存不可接单时段
  说明：
    · 商家选择一个项目 + 日期
    · 系统根据 open_time / close_time 按 30 分钟粒度生成时段网格
    · 商家点击红色标记不可接单的时段
    · 保存后，C端用户将无法选择这些时段
  ┌───────────────────┐
  │ unavailable_slots │  id / m_id / studio_id / disabled_date /
  │ 表                │  time_slot(VARCHAR 8, HH:MM) / time_slot_end /
  │                   │  created_at
  └───────────────────┘

步骤 1.5  其他设置 (可选)
  ─────────────────
  接口：POST /api/settings
  可设置：
    · shopName       — 店铺名称
    · shopMode       — 营业模式
    · announcement   — 公告文字 (显示在C端首页顶部)
    · qrCodeUrl      — 通用收款码
    · alipayQrUrl    — 支付宝收款码
    · wechatQrUrl    — 微信收款码

================================================================================
  第二阶段：C端用户预约下单流程
================================================================================

前提条件：
  · 用户通过商家分享的链接进入: https://xxx.com/?mId=shop_xxx
  · 浏览器 localStorage 自动生成 deviceId 作为用户标识
  · 整个过程无需注册/登录

步骤 2.1  浏览项目列表
  ─────────────────
  接口：GET /api/settings?mId=xxx      — 获取商家公告、收款码
        GET /api/studios-lite?mId=xxx  — 获取项目精简列表
  前端显示：
    · 顶部公告栏 (announcement)
    · 项目卡片列表 (封面图 + 名称 + 价格 + 包天/计费标签 + 定金比例)
    · 空状态提示 (无项目时)

步骤 2.2  进入项目详情
  ─────────────────
  用户点击某项目卡片 → 进入详情页 (currentView = 'detail')
  显示信息：
    · 项目详情大图 (detailImgUrls)
    · 项目标题 + 描述
    · 服务模式 / 计费方式
    · 营业时间 (openTime ~ closeTime)
    · 排期间隔 (timeInterval)
    · 起订张数 / 额外人数费

步骤 2.3  选择预约日期 (★ 核心交互)
  ─────────────────
  ┌─────────────────────────────────────────────────────────────────┐
  │  日历组件交互流程                                                │
  │                                                                  │
  │  ① 日历初始化                                                    │
  │     · 显示当月日历 (7列 × 6行 网格)                              │
  │     · 表头: 日 一 二 三 四 五 六                                 │
  │     · 可通过 ◀ 上月 / 下月 ▶ 切换月份                            │
  │                                                                  │
  │  ② 日期状态标记                                                  │
  │     · 过去日期 → 置灰不可选                                      │
  │     · 今日     → 蓝色高亮边框                                    │
  │     · 节假日   → 红色文字标记 (通过 timor.news API 查询)         │
  │     · 有订单日期 → 显示红点 (通过 /my-orders 获取)               │
  │                                                                  │
  │  ③ 用户点击某个可用日期                                          │
  │     → 触发 selectDate(dateStr)                                   │
  │     → 清空已选时段、关闭包天开关                                  │
  │     → 调用 GET /booked-times 获取该日占用情况                     │
  └─────────────────────────────────────────────────────────────────┘

步骤 2.4  时段选择与实时价格计算 (★ 核心交互)
  ─────────────────
  ┌─────────────────────────────────────────────────────────────────┐
  │  时段网格交互流程                                                │
  │                                                                  │
  │  ① 生成时段网格                                                  │
  │     · 根据项目的 openTime / closeTime / timeInterval 生成时段列表│
  │     · 例如: openTime=09:00, closeTime=18:00, interval=60       │
  │       生成: 09:00  10:00  11:00  12:00  13:00  14:00           │
  │             15:00  16:00  17:00                                │
  │                                                                  │
  │  ② 时段状态标记                                                  │
  │     · 已占用时段 (slot_bookings) → 灰色 × 不可选                 │
  │     · 不可接单时段 (unavailable_slots) → 灰色 × 不可选           │
  │     · 可用时段 → 白色 × 可点击                                   │
  │     · 已选时段 → 蓝色 ✓ 高亮                                     │
  │                                                                  │
  │  ③ 用户点击时段 → toggleTime(slot)                               │
  │     · 再次点击取消选择                                            │
  │     · 支持多选 (可连选多个连续或不连续时段)                       │
  │     · 选中后自动排序                                              │
  │                                                                  │
  │  ④ 包天模式切换                                                  │
  │     · 仅在项目 isPackageDay=true 时显示开关                      │
  │     · 开启包天 → 跳过时段选择，全天可用                           │
  │     · 关闭包天 → 恢复时段选择                                     │
  └─────────────────────────────────────────────────────────────────┘

步骤 2.5  实时价格计算
  ─────────────────
  每次选择/取消时段或修改参数后触发 calcPrice()
  ┌─────────────────────────────────────────────────────────────────┐
  │  价格计算逻辑 (前端 + 后端 priceEngine 一致):                     │
  │                                                                  │
  │  计费模式 = time (按时段)                                        │
  │    ① 确定单价:                                                   │
  │       · 节假日不同价 + 当天是节假日 → 使用 holidayPrice          │
  │       · 节假日不同价 + 当天是工作日 → 使用 weekdayPrice          │
  │       · 否则                       → 使用 price (统一价)        │
  │    ② 多时段折扣匹配:                                             │
  │       · 按 hours 从大到小排序 discountRules                      │
  │       · 找到第一个 hours >= 所选时段数的规则                     │
  │       · 命中则单价替换为该规则的 price                           │
  │    ③ 基础价格 = 时段数 × 最终单价                                │
  │                                                                  │
  │  计费模式 = photo (按张)                                         │
  │    ① 基础价格 = 拍摄张数 × 单价                                  │
  │                                                                  │
  │  计费模式 = photo_time (按张+时段)                               │
  │    ① 基础价格 = 拍摄张数 × 单价                                  │
  │    ② 校核: 每时段最大张数上限                                    │
  │                                                                  │
  │  包天模式                                                        │
  │    ① 基础价格 = packagePrice (包天价)                            │
  │                                                                  │
  │  额外人数费 (所有模式通用):                                       │
  │    extraTotal = (人数 - 1) × extraPersonFee × 时长(or张数)       │
  │                                                                  │
  │  总价 = 基础价格 + 额外人数费                                    │
  │  定金 = 总价 × depositRatio / 100                                │
  └─────────────────────────────────────────────────────────────────┘

步骤 2.6  填写预约信息
  ─────────────────
  需填写的表单字段：
    · photoCount  — 拍摄张数 (按张计费时显示, 默认 minPhotos)
    · peopleCount — 人数 (默认 1, 影响额外人数费)
    · contactNote — 联系方式 + 备注 (必填, 标签名由 customRemarkLabel 决定)
  前端校验：
    · 必须选择至少 1 个时段 (包天模式除外)
    · contactNote 不能为空
    · 防重复提交 (submitting 锁)

步骤 2.7  提交订单 (★ 核心步骤)
  ─────────────────
  接口：POST /api/create-order
  提交数据：
    ┌──────────────────────────────────────────────────┐
    │ mId             — 商家 m_id                       │
    │ studioId        — 项目 ID                         │
    │ studioTitle     — 项目名称                        │
    │ serviceMode     — 服务模式                        │
    │ date            — 预约日期 (YYYY-MM-DD)            │
    │ times[]         — 时段数组 ["09:00","10:00"]      │
    │ timesEnd[]      — 结束时段数组 (可选)              │
    │ isPackageOrder  — 是否包天                         │
    │ totalPrice      — 总价 (前端计算)                  │
    │ depositAmount   — 定金金额 (前端计算)              │
    │ depositRatio    — 定金比例                         │
    │ contact         — 联系方式+备注                    │
    │ peopleCount     — 人数                             │
    │ photoCount      — 拍摄张数                         │
    │ userDeviceId    — 用户设备ID                       │
    │ calcMode        — 计费方式                         │
    │ isHolidayPrice  — 是否节假日不同价                 │
    └──────────────────────────────────────────────────┘

  后端处理 (事务内，顺序执行):
    ┌─────────────────────────────────────────────────────────────┐
    │ ① 校验项目是否存在 (studioRepo.findByIdAndMerchant)         │
    │                                                             │
    │ ② 检查包天冲突                                              │
    │    · 查询 slot_bookings 该日是否有包天占用                   │
    │    · 若有 → 返回错误 "该日已被包场锁定"                      │
    │                                                             │
    │ ③ 检查不可接单时段                                          │
    │    · 查询 unavailable_slots                                 │
    │    · 若所选时段在不可接单列表中 → 返回错误                   │
    │                                                             │
    │ ④ 逐时段检查冲突 (行级锁 SELECT ... FOR UPDATE)              │
    │    · 查询 slot_bookings 同时段是否被占                       │
    │    · photo_time 模式: 检查该时段累计张数是否超上限           │
    │    · 若有冲突 → 回滚事务，返回错误                           │
    │                                                             │
    │ ⑤ 写入时段占用 (slot_bookings)                              │
    │    · 每个时段插入一行:                                       │
    │      m_id / studio_id / booking_date / time_slot /           │
    │      time_slot_end / order_no / is_package_day /             │
    │      booking_count                                          │
    │                                                             │
    │ ⑥ 写入订单 (orders)                                         │
    │    · 生成唯一 order_no (格式: MG + 时间戳 + 随机串)          │
    │    · 状态初始为 "待支付"                                     │
    │    · time_slots 以 JSON 存储                                │
    │                                                             │
    │ ⑦ 异步推送通知                                              │
    │    · 写入 notifications 表                                   │
    │    · WebSocket 实时推送给商家                                │
    └─────────────────────────────────────────────────────────────┘

  成功返回：{ orderNo, totalPrice, depositAmount }
  前端处理：
    · 关闭下单页 → 显示支付弹窗
    · 刷新我的订单列表

================================================================================
  第三阶段：支付流程
================================================================================

订单状态机：
  ┌────────┐   C端支付定金      ┌──────────┐   C端支付尾款     ┌──────────┐
  │ 待支付  │ ───────────────→ │ 已付定金  │ ─────────────→  │尾款待确认 │
  └───┬────┘                   └────┬─────┘                  └────┬─────┘
      │                             │                             │
      │ C端申请取消                  │ C端申请退款                  │ B端确认
      │ (仅待支付状态)               │                             │
      ▼                             ▼                             ▼
  ┌────────┐                   ┌──────────┐                  ┌────────┐
  │ 已取消  │                   │退款审核中 │                  │ 已结清  │
  └────────┘                   └────┬─────┘                  └───┬────┘
                                    │                            │
                              B端同意│     B端拒绝               │ B端归档
                                    ▼         ▼                  ▼
                              ┌────────┐ ┌──────────┐      ┌──────────┐
                              │已退款取消│ │ 回到     │      │已完成拍摄 │
                              └────────┘ │ 已付定金  │      └──────────┘
                                         └──────────┘

步骤 3.1  C端支付定金
  ─────────────────
  触发条件：订单状态 = "待支付"
  接口：POST /api/pay-deposit
  参数：{ orderNo, mId }
  后端处理：
    · 校验订单状态必须为 "待支付"
    · 更新订单状态 → "已付定金"
    · 注意：此为"确认已转账"操作，不涉及真实支付网关
  前端显示：
    · 弹窗展示: 应付定金金额 / 商家收款码 (支付宝+微信)
    · 用户线下转账后点击"我已支付定金"
  状态变更：待支付 → 已付定金

步骤 3.2  C端支付尾款
  ─────────────────
  触发条件：订单状态 = "已付定金"
  接口：POST /api/pay-final
  参数：{ orderNo, mId }
  后端处理：
    · 校验订单状态必须为 "已付定金"
    · 更新订单状态 → "尾款待确认"
  状态变更：已付定金 → 尾款待确认

步骤 3.3  B端确认结清
  ─────────────────
  触发条件：订单状态 = "尾款待确认"
  接口：POST /api/update-status
  参数：{ orderNo, status: "已结清", mId }
  后端处理：
    · 校验订单状态允许变更
    · 更新订单状态 → "已结清"
  状态变更：尾款待确认 → 已结清

步骤 3.4  B端完成拍摄 (归档)
  ─────────────────
  触发条件：订单状态 = "已结清"
  接口：POST /api/archive-order
  参数：{ orderNo, type: "已完成拍摄", mId }
  后端处理：
    · 校验订单归属
    · 更新订单状态 → "已完成拍摄"
  状态变更：已结清 → 已完成拍摄
  说明：此时订单进入"已完成"归档，不再出现在进行中列表

================================================================================
  第四阶段：取消与退款流程
================================================================================

步骤 4.1  C端直接取消 (仅 "待支付" 状态)
  ─────────────────
  触发条件：订单状态 = "待支付"
  C端操作：
    · 点击"取消"按钮 → 确认弹窗
    · 调用 POST /api/update-status
    · { orderNo, status: "已取消", mId }
  后端处理：
    · 校验状态为"待支付"
    · 更新状态 → "已取消"
    · 释放 slot_bookings 中该订单的时段占用
  状态变更：待支付 → 已取消 (终止)

步骤 4.2  C端申请退款 ("已付定金" 及之后状态)
  ─────────────────
  触发条件：订单状态不是 "待支付" 且不是终止状态
  接口：POST /api/apply-cancel
  需填写信息：
    · refundText   — 退款账号信息 (支付宝/微信账号等)
    · refundImgUrl — 收款码图片 (上传后获得URL)
  后端处理：
    · 更新 orders 表: refund_text / refund_img_url
    · 状态 → "退款审核中"
    · 通知商家 (notifications + WebSocket)
  状态变更：已付定金(或后续状态) → 退款审核中

步骤 4.3  B端处理退款申请
  ─────────────────
  B端同意 (POST /api/refund/approve):
    · 校验状态为"退款审核中"
    · 更新状态 → "已退款取消"
    · 释放 slot_bookings 时段占用
    · 通知用户

  B端拒绝 (POST /api/refund/reject):
    · 校验状态为"退款审核中"
    · 可填写拒绝原因 (reason)
    · 更新状态 → 回到"已付定金"
    · orders.reject_reason 记录拒绝原因
    · 通知用户

================================================================================
  第五阶段：改期流程
================================================================================

步骤 5.1  C端申请改期
  ─────────────────
  接口：POST /api/reschedule/apply
  需填写信息：
    · orderNo     — 订单号
    · oldDate     — 原预约日期
    · oldTimes[]  — 原时段数组
    · oldTimesEnd[] — 原结束时段数组 (可选)
    · newDate     — 新预约日期
    · newTimes[]  — 新时段数组
    · newTimesEnd[] — 新结束时段数组 (可选)
    · reason      — 改期原因 (选填)
  后端处理：
    · 写入 reschedule_requests 表
    · 状态初始为 "待处理"
    · 通知商家
  ┌─────────────────────┐
  │ reschedule_requests │  id / m_id / order_no / old_date /
  │ 表                  │  old_times(JSON) / old_times_end(JSON) /
  │                     │  new_date / new_times(JSON) /
  │                     │  new_times_end(JSON) / reason /
  │                     │  status('待处理'|'已同意'|'已拒绝') /
  │                     │  created_at / updated_at
  └─────────────────────┘

步骤 5.2  B端处理改期申请
  ─────────────────
  B端同意 (POST /api/reschedule/approve):
    · 事务内执行:
      ① 释放旧时段 (DELETE slot_bookings WHERE order_no=xxx)
      ② 检查新时段是否冲突 (SELECT ... FOR UPDATE)
      ③ 占用新时段 (INSERT slot_bookings)
      ④ 更新 orders: order_date / time_slots / time_slots_end
      ⑤ 更新 reschedule_requests.status → "已同意"
      ⑥ 通知用户

  B端拒绝 (POST /api/reschedule/reject):
    · 更新 reschedule_requests.status → "已拒绝"
    · 可选填写拒绝原因
    · 通知用户
