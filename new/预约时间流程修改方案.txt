================================================================================
  喵喵自助下单平台 — 升级版完整业务流程（五窗口高精度排期系统）
  从建立项目(3窗口) → 用户分步预约(2窗口) → 支付 → 拍摄完成 → 结清尾款
================================================================================

  平台角色说明： [cite: 1]
    B端 = 商家管理后台 (admin.html)，需 JWT 登录 [cite: 1]
    C端 = 用户预约端 (index.html)，通过 URL 参数 ?mId=xxx 访问，无需登录 [cite: 1]

================================================================================
  第一阶段：商家注册与登录 (基础模块)
================================================================================

步骤 1.1  商家注册 [cite: 1]
  ─────────────────
  接口：POST /api/register [cite: 1]
  需填写信息：username(登录账号)、password(登录密码)、shopName(店铺名称) [cite: 1]
  后端处理：生成 m_id (格式: shop_${时间戳})，bcrypt哈希密码，写入 merchants 表，返回 JWT Token [cite: 2, 3]。

步骤 1.2  商家登录 [cite: 3]
  ─────────────────
  接口：POST /api/login [cite: 3]
  后端处理：验证密码，签发 JWT Token (有效期 2h)，前端存入 localStorage [cite: 3]。

================================================================================
  第二阶段：商家项目配置与上架 (B端 — 3个独立跳转窗口)
================================================================================

【窗口 1：项目基础信息窗口】(商家录入基本盘)
  ─────────────────────────────────────────────────────────────────
  操作：商家点击“上架新项目”，系统弹起或跳转至一个全新的独立窗口。
  接口：POST /api/add-studio/step1 (需 JWT 认证)
  需填写与配置信息：
    · title              — 项目名称 (如 "Coser专属主题摄影") [cite: 5]
    · city               — 地点城市 (如 "上海")
    · availableDates[]   — 可选日期数组 (通过日历组件多选或框选多个日期)
  交互逻辑：
    · 商家填写完毕后，点击窗口底部的“下一步”按钮。
    · 系统校验必填项，校验通过后自动关闭当前窗口，并【跳转至窗口 2】。

【窗口 2：高精度工作时间与休息排期窗口】(生成一日接单总时间轴)
  ─────────────────────────────────────────────────────────────────
  操作：从窗口 1 跳转而来，承接独立的输入交互。
  接口：POST /api/add-studio/step2 (需 JWT 认证)
  需填写与配置信息：
    · baseWorkTime       — 基础不间断工作时间段 (包含起始与结束时间，如 09:00 至 21:00)
    · restSlotCount      — 休息分段数 (输入数字 N，如输入 2)
    · restSlots[]        — 休息分段起止时间 (根据输入的 N 动态生成 N 行起止输入框)
                           例: 若 N=2，输入 [ {start:"12:00", end:"13:00"}, {start:"18:00", end:"18:30"} ]
    · intervalRestTime   — 每单结束后的间隔休息时间 (分钟数，如 15 分钟，用于转场调备)
    · isReplicated       — 复刻选项 (勾选框："是否复刻至后续选定日期")
  核心后端处理与计算逻辑：
    · 系统计算该日“可接单时间”总集 = [baseWorkTime] 扣除所有的 [restSlots]。
    · 若勾选“复刻选项”，则系统会自动将此一日的工作与休息规则，批量复刻至【窗口 1】中所选取的所有可选日期中。
  交互逻辑：
    · 商家填写完毕后，点击窗口底部的“下一步”按钮。
    · 数据暂存，系统自动关闭当前窗口，并【跳转至窗口 3】。

【窗口 3：接单样式与服务时长定价窗口】(定义商品属性与时间开销)
  ─────────────────────────────────────────────────────────────────
  操作：从窗口 2 跳转而来，进行上架前的最终商品属性配置。
  接口：POST /api/add-studio/step3 (需 JWT 认证)
  需填写与配置信息：
    · styles[]           — 样式管理列表，支持添加多个条目：
                           ├─ styleName     : 样式名称 (如 "古风常服" / "机能赛博")
                           └─ styleCoverUrl : 风格示意图 (单张风格大图，上传获得)
    · priceSettings      — 价格设置：
                           ├─ singlePrice   : 单张价格 (元)
                           ├─ hasPackage    : 是否加入套餐 (true/false 开关)
                           └─ packagePrice  : 套套价格 (启用套餐时有效)
    · durationSettings   — 标准拍摄时间配置：
                           ├─ singleShotTime: 单张平均拍摄时间 (分钟数，如 10 分钟)
                           └─ packageTime   : 套餐拍摄时间 (分钟数，如 90 分钟)
    · experienceSettings — 模特经验与特殊加时配置：
                           ├─ isNoviceEnabled : 是否让顾客选择模特经验 (true/false 商家开关)
                           ├─ noviceSingleAdd : 新人单张加时 (分钟数，如额外加 5 分钟)
                           └─ novicePackageAdd: 新人套餐加时 (分钟数，如额外加 20 分钟)
    · depositRatio       — 定金比例设置 (5% / 10% / 30% / 50% / 100%) [cite: 17]
  交互逻辑：
    · 商家确认无误后，点击底部的“完成并上架项目”按钮。
    · 后端整合 3 个窗口的数据写入主表（studios、studio_styles、studio_schedules），保存成功后关闭窗口，返回项目列表。

================================================================================
  第三阶段：C端用户分步预约流程 (C端 — 2个独立跳转窗口)
================================================================================

前提条件：
  · 用户通过商家分享的链接进入（携带 mId），无需注册登录 [cite: 1]。
  · 浏览项目精简列表及详情后 [cite: 23, 24]，点击“立即预约”，进入核心分步预约窗口。

【窗口 4：客户样式选择与时间段预估窗口】(高精度时间占用动态精算)
  ─────────────────────────────────────────────────────────────────
  操作：客户点击预约后弹起的第一个独立大窗口。
  接口：GET /api/studio-schedules/available (动态查询可用起点)
  用户输入与选择项：
    · optType            — 预约类型选择 ("按单张" 或 "选特定套餐")
    · photoCount         — 拍摄张数 (选择"按单张"时可见，输入张数)
    · selectedStyle      — 选择样式 (展现商家在【窗口3】上传的样式名称与风格示意图)
    · modelExperience    — 模特身份选择 (仅在商家端开启时展示：可选 "新人" 或 "有经验的模特")
    · bookingStartDate   — 拍摄预约日期 (选择商家开放的可选日期)
    · bookingStartTime   — 拍摄起始时间点 (用户在可选时间轴中点击选择一个开始点，如 14:00)
  前端实时计算与动态展示机制：
    · 每次用户修改上述任意参数，前端或后端计算引擎立即动态演算【预计拍摄时间段】：
      1. 计算基础时长：若选单张 = 张数 × singleShotTime；若选套餐 = packageTime。
      2. 计算身份加时：若选新人且商家开启，单张叠加 (张数 × noviceSingleAdd)，套餐叠加 novicePackageAdd。
      3. 计算总占用时长 = 基础时长 + 身份加时 + 每单结束后间隔休息时间(intervalRestTime)。
    · 界面显著位置实时向客户展示：【预计拍摄时间段：14:00 - 15:15 (已包含转场休息)】
    · 校验冲突：系统自动校验该区间是否与当天的其他订单冲突，或是否侵入了商家设置的“休息分段时间”。
  交互逻辑：
    · 客户确认预估的时间段满意后，点击“下一步”按钮，当前窗口关闭并【跳转至窗口 5】。

【窗口 5：角色信息完善与双重锁死提交窗口】(双锁核心逻辑)
  ─────────────────────────────────────────────────────────────────
  操作：从窗口 4 跳转而来，进行最后的下单冲刺。
  接口：POST /api/create-order
  需填写信息：
    · roleName           — 想要拍摄的角色名称 (必填)
    · contactNote        — 联系方式与拍摄备注 (选填)
  
  核心【双重时间锁定】流转机制：
    1. 用户端预锁定 (定金锁)：
       · 客户在此窗口点击“提交订单并前往支付定金”。
       · 订单生成后状态为“待支付” [cite: 72]。客户在线下转账或通过收款码支付定金并点击“我已支付” [cite: 84]。
       · 接口 POST /api/pay-deposit 触发：【用户端的预定时间段被锁死】。在C端日历和排期轴上，该时间段立即对其他任何浏览者置灰，进入预锁定保护状态，防止并发抢单。
    2. 摄影师端最终锁定 (确认锁)：
       · 商家（摄影师）的 B端后台通过 WebSocket 收到实时新订单通知 [cite: 74, 115]。
       · 摄影师在订单列表中查看到这笔订单的时间区间、角色名称和备注。
       · 摄影师手动核对并点击“确认接单”：
         接口：POST /api/order/confirm-lock
         后端正式将该订单的时间记录写入底层的 `slot_bookings` 核心表中并【彻底锁死】。
       · 订单状态流转，且该订单正式正式加入摄影师的“进行中工作订单列表”。
  交互逻辑：
    · 双锁完成后，当前窗口关闭，提示用户“预约成功，等待拍摄”。

================================================================================
  第四阶段：后续常规状态机流程 (支付尾款、取消、退款) [cite: 76]
================================================================================

  订单完整状态流转参考： [cite: 76]
    待支付 -> 已付定金(用户锁) -> 尾款待确认 -> 已结清(摄影师归档) -> 已完成拍摄 [cite: 76, 84, 85]。
    
  释放时间轴规则：
    · 处于“待支付”若C端点击取消，立即释放时间轴 [cite: 86]。
    · 处于“已付定金”及之后状态，C端需申请退款，仅在 B端商家点击“同意退款”后，系统才在事务中正式释放该段占用 [cite: 86, 87]。
